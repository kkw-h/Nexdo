# Nexdo AI 指令模块开发规划

## 1. 目标

本模块用于把用户的自然语言指令转换为 Nexdo 内可执行的业务动作，并确保所有会修改数据的动作都必须经过前端确认后才能真正执行。

当前规划聚焦以下目标：

- 支持提醒相关自然语言操作
- 支持单动作和多动作组合指令
- 让 AI 负责理解意图和生成执行建议
- 让 Go 后端负责鉴权、数据查询、确认校验和最终执行
- 让 Flutter 前端负责展示候选结果、让用户确认、反馈执行结果

## 2. 总体原则

- `ai-service` 不直接访问数据库，不直接执行写操作
- `server` 是前端唯一调用入口
- 任何创建、修改、删除、完成类动作都必须先确认
- AI 返回的是“建议执行计划”，不是最终执行结果
- 多动作按顺序执行，保证流程可解释、可回放、可审计

## 3. 模块边界

### 3.1 `app/`

负责：

- 接收用户自然语言输入
- 调用 `server` 的 AI 指令接口
- 展示 AI 识别出的意图、命中的目标数据、待执行动作
- 对需要修改数据的动作弹出确认
- 用户确认后调用执行接口
- 展示执行结果和失败原因

### 3.2 `server/`

负责：

- 用户鉴权
- 根据用户输入构建 AI 上下文
- 查询提醒、清单、分组、标签等候选数据
- 调用 `ai-service` 做意图识别和计划生成
- 校验 AI 返回动作是否合法
- 持久化确认记录和一次性确认令牌
- 执行最终业务动作
- 记录执行结果并防止重复执行

### 3.3 `ai-service/`

负责：

- 识别用户意图
- 将自然语言转成结构化计划 `plan`
- 约束输出格式
- 对模糊描述返回候选解释或澄清结果

不负责：

- 数据写入
- 权限判断
- 最终业务校验
- 执行数据库操作

## 4. 目标流程

标准流程定义为：

1. 用户输入自然语言
2. `server` 识别请求上下文并查询候选数据
3. `server` 调用 `ai-service` 生成结构化执行计划
4. `server` 校验计划中的动作类型、目标对象、补丁字段
5. 如果涉及写操作，`server` 生成确认记录和确认令牌
6. `app` 展示待执行动作给用户确认
7. 用户确认后，`app` 调用执行接口
8. `server` 校验确认令牌未过期、未消费、归属当前用户
9. `server` 按顺序执行计划
10. 返回执行结果给前端

## 5. 指令拆解模型

为支持“取消今天所有提醒，并添加晚上九点的约会提醒”这类复合表达，规划统一采用 `plan` 模型。

### 5.1 为什么使用 `plan`

- 单动作和多动作可以共用一套协议
- 前端展示确认信息更简单
- 后端执行顺序明确
- 失败时更容易定位是哪个步骤出错

### 5.2 结构约定

每条用户指令最终都被归一化为：

```json
{
  "plan": [
    {
      "action": "delete_reminder",
      "reason": "用户要求取消今天所有提醒",
      "targets": []
    },
    {
      "action": "create_reminder",
      "reason": "用户要求新增晚上九点的约会提醒",
      "patch": {}
    }
  ]
}
```

其中每个步骤至少包含：

- `action`
- `reason`
- `targets` 或 `patch`

## 6. 当前可执行动作

当前规划与已实现动作保持一致：

- `create_reminder`
- `update_reminder`
- `delete_reminder`
- `complete_reminder`
- `uncomplete_reminder`

后续可扩展动作：

- `create_quick_note`
- `convert_quick_note_to_reminder`
- `move_reminder_to_list`
- `set_reminder_tags`
- `batch_update_reminders`

## 7. 数据确认策略

### 7.1 必须确认的情况

以下情况必须返回前端确认：

- 新增提醒
- 修改提醒标题、备注、时间、列表、分组、标签
- 删除提醒
- 批量完成或取消完成提醒
- 任何多动作组合指令

### 7.2 可直接返回结果的情况

仅查询类能力未来可考虑跳过确认，例如：

- “今天有哪些提醒”
- “有哪些已完成提醒”
- “下周的任务清单是什么”

当前阶段本模块先聚焦写操作链路。

### 7.3 确认记录设计

确认记录需要持久化以下信息：

- 用户 ID
- 原始输入
- 结构化动作或 `plan`
- 目标 ID 列表
- 补丁内容
- 过期时间
- 是否已消费
- 执行结果摘要

确认令牌必须满足：

- 单次使用
- 归属用户校验
- 过期校验
- 重放拦截

## 8. API 规划

### 8.1 `server`

当前服务端规划以以下接口为核心：

- `POST /api/v1/ai/commands/resolve`
  - 输入自然语言
  - 返回 AI 生成的结构化计划和确认信息
- `POST /api/v1/ai/commands/confirmations/verify`
  - 校验确认记录是否仍可执行
- `POST /api/v1/ai/commands/confirmations/execute`
  - 在确认后执行计划

### 8.2 `ai-service`

当前 AI 服务规划以以下接口为核心：

- `POST /api/commands/classify`
  - 输出意图分类结果
- `POST /api/commands/propose`
  - 输出结构化动作建议或多动作 `plan`

## 9. 开发阶段划分

### 阶段一：基础链路

- 建立 `ai-service` 基础服务
- 定义 prompt 系统和结构化输出协议
- 打通 `server -> ai-service` 调用
- 实现提醒领域单动作解析
- 前端支持确认后执行单动作

验收标准：

- 能处理新增、修改、删除、完成、取消完成提醒
- 所有写操作都要求确认
- 执行链路具备基本错误处理

### 阶段二：多动作执行

- 引入统一 `plan` 模型
- 支持一条指令对应多个步骤
- 后端按顺序执行多个动作
- 前端按步骤展示确认信息

验收标准：

- 支持复合表达
- 支持批量目标匹配
- 单步骤失败时返回明确失败位置

### 阶段三：数据命中与澄清

- 增强候选数据查询能力
- 支持基于时间、标题、列表、标签等条件命中提醒
- 对歧义目标返回澄清结果

验收标准：

- 能处理“今天所有提醒”“工作清单里未完成提醒”等表达
- 目标不唯一时不会误执行

### 阶段四：前端交互优化

- 对多动作确认做卡片化展示
- 展示执行前后差异
- 对失败步骤提供重试或手动修正入口

验收标准：

- 用户可清楚看到将要发生的修改
- 出错时具备可恢复路径

### 阶段五：能力扩展

- 扩展到闪念、清单、分组、标签
- 引入更细粒度权限和风险控制
- 增加审计日志和指标监控

## 10. 当前阶段已落地内容

当前已完成：

- monorepo 新增 `ai-service/`
- 建立 prompt system 与 command system
- `ai-service` 支持 `classify` 和 `propose`
- `server` 支持 AI 指令解析、确认校验、确认执行
- 确认记录持久化与一次性令牌机制
- 支持 `plan` 多动作顺序执行
- 已覆盖提醒创建、修改、删除、完成、取消完成

## 11. 下一步建议

建议优先继续推进以下事项：

- 在 `app/` 中补齐 AI 指令输入入口和确认页
- 增加歧义目标澄清协议，而不是依赖 AI 猜测
- 为多动作执行补充部分成功和回滚策略定义
- 增加操作审计日志
- 增加更多集成测试，覆盖批量删除、批量完成、时间解析等边界情况

## 12. 风险与约束

- AI 可能误判意图，因此写操作必须确认
- 候选数据不足时，AI 计划可能不稳定，因此后端必须二次校验
- 多动作执行存在部分成功风险，需要前端明确展示结果
- 时间语义依赖用户时区，后端构造上下文时必须带上时区信息

## 13. 结论

本模块的核心不是“让 AI 直接操作数据”，而是“让 AI 生成可验证、可确认、可执行的结构化计划”。

只要继续坚持这条边界，后续无论是扩展到更多业务对象，还是增加更复杂的自然语言组合指令，整体架构都还能稳定演进。
